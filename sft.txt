本文件用于压缩当前 Echo 项目复现对话上下文。原则：较早内容做摘要；最近 10 个来回尽量完整保留，以便后续继续讨论 SFT、interleaved inference、GRPO、VERL/vLLM 等问题。

## 一、较早对话摘要

### 1. 项目主线

当前项目目标是复现 Echo: Towards Advanced Audio Comprehension via Audio-Interleaved Reasoning。核心思想是让 Qwen2.5-Omni-7B 从普通 audio-conditioned text reasoning 进化为 audio-interleaved reasoning：模型在推理中生成 `<seg>start,end</seg>`，系统暂停生成，裁剪原始音频对应片段，将音频片段插回上下文，再继续生成。

论文主流程被理解为：

```text
带时间标注的音频数据
-> Qwen2.5-Omni 提取音频文本信息 A1/A2/A3
-> 结合强标注时间元数据 A4/A5
-> DeepSeek-R1 生成 QA-CoT
-> DeepSeek-R1 重评 QA valid / COT valid
-> 根据重评结果分成 EAQA-SFT / EAQA-RL
-> SFT 得到 cold-start model
-> 推理时启用 audio-interleaved inference
-> 使用 EAQA-RL 做 RL / GRPO
```

论文分流逻辑：

```text
QA valid = Yes 且 COT valid = Yes -> SFT
QA valid = Yes 且 COT valid = No  -> RL，去掉 CoT
QA invalid                         -> 丢弃
```

### 2. 已完成的数据与 SFT 工作

项目已经完成：

- AudioSet-Strong parquet -> flac + metadata jsonl。
- A1/A2/A3 生成：音频描述、语音信息、音乐信息，使用本地 Qwen2.5-Omni。
- QA skeleton 构建：`start_percentage`、`duration_percentage`、`gap`、`overlap`、`count_before`、`repeated_event_gap`、`duration_compare`、`order`。
- 早期 SFT 数据混合生成：DeepSeek-R1 润色 + 本地模板。
- strict SFT jsonl 清洗：保留 messages/audios，检查 `<think>`、`<answer>`、`<seg>`、answer in choices、去标签污染、去重。
- Qwen2.5-Omni LoRA SFT：ms-swift、lr=5e-6、fp16、QWEN_OMNI_SKIP_SPK=1。
- 固定评估集 eval_manifest_500 和 eval_from_manifest.py。

早期评估：

```text
v7: has_seg ≈ 60%, answer_acc ≈ 30%
v8: has_seg ≈ 27.4%, answer_acc ≈ 44.6%
v9a-format-clean: has_seg ≈ 39.4%, answer_acc ≈ 42.0%, answer_in_choices 降到 87%
v9b-diverse-cot-2epoch: has_seg ≈ 99.8%, answer_acc ≈ 43.8%，目前作为较好 cold-start 候选
```

### 3. 数据 judge 与过滤

实现了 OpenAI-compatible / 第三方中转的 DeepSeek-R1 judge 脚本，支持 base_url、model、api_key_env、resume、retry_failed。20 条 smoke test 全成功，无 API 错误和解析错误。

随后做了 600 条按 source_group + type 分层的 judge：

```text
total=600
qa_valid_yes=542, qa_valid_no=58
cot_valid_yes=501, cot_valid_no=99

```

按题型结论：

```text

duration_percentage / order 有一定噪声

```

后续建立了四类数据分流：SFT、RL、discard、needs_review_or_rewrite。`start_percentage` 和 `deepseek_polished + overlap` 暂时进入 needs_review_or_rewrite，而不是永久丢弃。

### 4. 关于 `<seg>` 后续写风格

检查官方 EAQA_SFT_100subset 后发现：官方 SFT 中 `</seg>` 后接大写字母或 `<` 的情况为 0，大量是：

```text
</seg>, ...
</seg> reveals ...
</seg>, meaning ...
</seg>, indicating ...
```

这说明 Echo 官方 SFT 数据天然教了模型“segment 后自然续写”，而不是新开一句或立刻接下一个标签。这个特征对后续 RL 很关键：如果 SFT 没有这种 continuation 风格，Rconsist 只能普遍扣分，GRPO 难以学到相对优势。

结论：

```text
SFT 教正确写法；
Rconsist 惩罚偏离写法；
RL 不是从零教 continuation。
```

### 5. VLM-R3 对 Echo 的启发

阅读了 VLM-R3：视觉领域的交错推理框架，做法是模型生成 bbox，外部系统裁剪图像、zoom、插回上下文，模型继续推理。

VLM-R3 流程：

```text
1. 构造 VLIR 数据集
2. 过滤 bbox 有效性和 reasoning 逻辑
3. VLIR SFT 得到 cold-start
4. interactive inference: bbox -> crop/zoom -> visual tokens -> continue
5. R-GRPO 强化
6. 主实验、消融、grounding/attention 分析
```

Echo 与 VLM-R3 对应关系：

```text
VLM-R3: {"bbox_2d": [...]} -> crop image -> insert visual tokens
Echo:   <seg>s,e</seg>      -> clip audio  -> insert audio tokens
```

### 6. MMAR / Qwen2.5-Omni baseline 评估问题

Qwen2.5-Omni 技术报告没有报告 MMAR，只报告 MMAU，MMAU Avg=65.60。Echo 论文中报告 Qwen2.5-Omni on MMAR = 57.33，Table 3 reproduced Base Model = 51.80。

本地 MMAR 自由生成评估发现：

```text
strict_acc ≈ 33.4%
fallback_acc ≈ 42.0%
has_answer_tag ≈ 70.5%
gibberish / pérdida 死循环约 15-22%
```

当前判断：本地评估和 Echo 论文 baseline 存在协议或实现差异。需要区分 strict/fallback、prompt、decoding、输入方式、是否 forced-choice/logprob 等。

### 7. GRPO 相关理解

Echo reward：

```text
R = Rformat + Rconsist + Racc + Rseg
```

- Rformat：封装格式正确给 0.5。
- Rconsist：`</seg>` 后下一个文本 token 若为大写或 `<`，每次 -0.1，最多 -0.5。
- Racc：答案匹配 ground truth 给 0.5。
- Rseg：答案正确且至少引用一个 segment 给 0.5。

GRPO 解释：reward 是 rollout-level，advantage 是 rollout-level；但 PPO-style ratio / KL 是 token-level，所以训练时仍需要每个生成 token 的 logprob。插入的 audio tokens 是环境注入，不是模型动作，必须参与上下文但不参与 loss。

严格版 interleaved GRPO 应该是：

```text
audio tokens: 参与 forward / attention / conditioning
audio tokens: loss mask = 0
text / <seg> / <answer>: loss mask = 1
```

当前工程中存在 text-only GRPO approximation：rollout 是 interleaved，但训练 forward/backward 只对 text prompt + final_response 算 logprob。这是可跑基线，但不等价于论文严格版。

### 8. GRPO 工程排查与当前状态

最初完整 batch forward 会在 Qwen2.5-Omni attention 中崩溃。排查发现是 policy forward with grad 的 `(B,T)` 过大。实验：

```text
bs=1/2/4 OK
bs>=8 crash
```

因此采用 policy_forward_micro_batch_size=4 的 micro-batch 方案，已通过 smoke test。

随后 rollout 阶段出现 CUDA device-side assert，进入子进程隔离模式。commit fcc5fdf 添加了 subprocess-isolated rollout workers，worker 子进程跑 interleaved rollouts，主进程做 text-only GRPO forward/backward。20 samples / 3 steps / 12 workers smoke test 通过。

VERL + vLLM 尝试：

- vLLM 0.12.0 绑定 torch 2.9.0。
- FSDP / DeepSpeed ZeRO-2 / ZeRO-3 都在初始化分布式或模型包装阶段 SIGSEGV。
- 当前判断：torch 2.9.0 + 当前集群 NCCL/distributed + Qwen2.5-Omni/VERL 组合不稳定。
- VERL 训练路线暂时不作为主线，除非做环境隔离：vLLM 只做 rollout server，VERL 在稳定 torch 环境中做训练。

### 回合 2

用户：

pool并行是在多张显卡上面跑还是在一张上面跑

助手：

推荐是在多张显卡上跑。不要默认在一张 GPU 上开多个 pool worker。每个 rollout worker 都要加载一份 Qwen2.5-Omni-7B + LoRA，单张 80GB 卡通常承受不了多个完整 worker，或者会显著抢显存、抢算力，反而更慢、更容易崩。

```text
rollout_worker_mode = persistent
num_rollout_workers = 1
```

多卡情况：

```text
rollout_worker_mode = pool
num_rollout_workers = N
worker_devices = "0,1,2,3"
```

每个 worker 绑定一张 GPU。若有多张 GPU，最好 GPU0 做训练，GPU1/GPU2 做 rollout worker。pool 是多卡并行方案；单卡只用 persistent worker。

### 回合 3

用户：

我想这个pool模式并不是我想要的,我想要的是单卡多卡都能跑, 最好是vllm或者ver那种架构的并行推理版本

助手：

你说得对：你想要的不是“多张 GPU 每张开一个 worker”的简单 pool，而是更像 vLLM / verl actor 架构的 rollout 服务化并行推理。

理想结构是：

```text
Trainer Process
  - sampling batch
  - reward
  - GRPO loss
  - backward / optimizer step

Rollout Engine / Rollout Server
  - 常驻加载 policy model
  - 接收多个 rollout request
  - 支持 batch / async / queue
  - 执行 interleaved generation
```

Echo 的特殊难点是 rollout 不是普通 prompt -> response，而是 generate 到 `<seg>` 后要暂停、裁剪音频、追加 audio、继续生成。因此建议先做自定义 persistent rollout server，不要直接接 vLLM。后端抽象：`HFBackend` 先实现，未来再加 `VLLMBackend`。

6 FSDP sdpa off P2P/IB disable+unset ROCR+CUDA_BLOCKING=0 SIGSEGV actor_rollout_wg.init_model()
7 DeepSpeed ZeRO-2 sdpa off default SIGSEGV deepspeed.initialize()
8 DeepSpeed ZeRO-3 sdpa off default SIGSEGV deepspeed.initialize()
核心结论：torch 2.9.0 的 NCCL 通信层有 bug，任何分布式框架（FSDP / DeepSpeed / ZeRO-2 / ZeRO-3）在初始化 NCCL collective 操作时都触发 SIGSEGV。模型本身加载、LoRA 都正常。问题 100% 在 torch 2.9.0 的分布式通信层。

助手：

vLLM 不是永远绑定 torch 2.9.0，但某个 vLLM 版本会强依赖某个 PyTorch 版本。当前 vLLM 0.12.0 强制依赖 torch==2.9.0，所以你这个环境等于是被 torch 2.9.0 绑死了。

你 8 次实验显示 FSDP、DeepSpeed ZeRO-2、ZeRO-3 都在 torch 2.9.0 环境下 SIGSEGV，说明问题更像 torch 2.9.0 + 当前 CUDA/NCCL/驱动/集群环境 + 多进程分布式初始化。

路线：

- 保留 vLLM 0.12.0 只做推理，不做分布式训练。
- 如要训练，使用稳定 torch 环境，避免 vLLM 0.12.0 强制 torch 2.9。
- 若要测试旧 vLLM + 旧 torch，必须新建独立环境，不污染主环境。
- 当前训练主线继续用自定义 GRPO。
